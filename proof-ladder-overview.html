Operating Model Defined Built

2026-04-02

The proof is not “can a human manually rescue the parser.” The proof is “does the system run in the intended way.” That intended way is now explicit in the epics, acceptance tickets, and architecture docs: claim a real failed email from the live system, generate the parser fix through same-run local inference on the claimed .eml, keep iterating until the relevant local parser test suite is 100% green, then have an operator apply, deploy, complete, and replay.

This target behavior is now written directly into the epics and acceptance tickets, and it also matches the bounded-iteration language in docs/evolution-loop.md and docs/pipeline-overview.html.

Shared Runtime, Per-Shop Programs Built

2026-04-03

The system is already shop-scoped at the graph layer. Each proof run selects one profile_id, materializes that shop's graph into a proof-local repo, runs that shop's parser suite, and deploys only that shop graph. The shared parser runtime is the interpreter; each shop graph is a separate program executed by that interpreter.

This means the architectural question is no longer whether shop behavior is isolated structurally. It is whether that isolation is sufficient operationally: one shop proof proves one shop program under the shared interpreter, not global stability across every shop.

The important boundary rules are:

  • a change to one shop graph should affect only that shop's execution
  • a successful shop proof demonstrates that shop program, not the whole fleet
  • if a needed capability cannot be expressed by the current graph primitives, that is a shared-runtime issue, not a shop issue
  • the next real complexity wall is within-shop variation: multiple real email families for one merchant competing inside the same graph

Architecture Enforcement Of Iterate-To-Green Built

2026-04-02

The intended behavior is architectural, not proof-only: the local evolution loop should keep working the claimed item until the relevant parser suite is fully green. The current docs already describe that bounded iteration model, and the orchestration now enforces it in both the staged proof runner and the bridge runtime.

Current implementation: when a pass still leaves tests red, the orchestrator launches another same-run evolution pass with the latest failing test output and extract result supplied as the next handoff. This continues until the suite is green or the bounded retry budget is exhausted.

Generic Runtime Gap Handoff Built

2026-04-03

The desired end state is still that shop-specific behavior lives in the graph. But if same-run inference discovers that the current generic runtime cannot execute the needed graph semantics, that is not something the proof loop should patch ad hoc. The run must stop, surface a generic runtime/platform gap, and hand that off for manual review and generalization outside the run.

Working rule: graph first when the generic runtime already supports the needed semantics; manual platform work only when inference exposes that the generic runtime is too weak. After that generic improvement lands, the shop should still be fixed through graph changes rather than shop-specific code.

The important discipline is that inference is allowed to make the assessment, but not to define the boundary by itself. During the run, the model already has the graph, the generated local repo, the shared runtime code, the failing shop tests, and the proof handoff. That means it can often say something useful like still graph-solvable or likely generic runtime gap. We should use that. But we should not treat that statement as final.

The operating rule is:

  • the shop proof failure shows the real claimed email is still failing
  • same-run inference may explain why the graph path appears exhausted and may draft the GitHub issue text
  • that result is a diagnosis candidate only
  • a matching minimal shared-runtime reproduction must also fail before the case is classified as confirmed manual platform work
  • if that shared-runtime reproduction does not fail, the proof loop should keep treating the issue as graph work, not platform work

This keeps the model useful without letting difficult graph work drift into a vague runtime gap label. The model can accelerate diagnosis and ticket creation; the shared-runtime reproduction is what makes the classification real.

Proof Integrity Guardrails Built

2026-04-02

The living rule is that replay success only counts if it comes from the normal proof-generated fix for that run. Manual parser/runtime/schema/test edits, hand-curated graph changes, reused proposals, reused inference logs, or stopping before the local suite is fully green are all invalid proof paths.

The recurring confusion has been called out explicitly in the epics so it cannot be quietly reinterpreted later.

Inference-Time Visibility Built

2026-04-03

The inference moments are the key driving element of this process, so visibility into live reasoning and tool calling is a requirement for improving the pipeline, not an optional debugging extra. Wrapper summaries are not enough; we need direct access to the native inference transcript so we can inspect what the model read, which tools it called, where it spent time, and why it converged or stalled.

Current state: this is now part of the artifact contract. New runs keep a direct Codex event transcript in the artifact bundle, and the fallback Claude path still copies the native .jsonl transcript when used, so inference-time reasoning and tool calls can be inspected after the run. The handoff context now also carries the proof-required fields, disallowed error prefixes, and the rule that the run must stop cleanly if same-run inference determines that the current generic runtime cannot support the required graph semantics.

Blank-Slate Baseline Allowed Built

2026-04-03

A contaminated online state does not make the proof stronger. If earlier manual deployments, partial runs, or stale proof artifacts complicate interpretation, the correct move is to reset to a blank baseline and rerun the proof chain cleanly.

This is now part of the proof discipline and it is live. The GitHub prepare phase can clear the remote profile graph before attempt 1, so the proof chain starts from a blank online baseline instead of inheriting earlier manual or proof-created state. The Decathlon prepare run 23929273000 demonstrated that the blank-slate delete and first failed attempt now work in the live system.

Goldens Encode Order Basics Built

2026-04-02

The proof-local green gate is not just schema smoke. It already includes the generated shop-local golden test. That matters because the golden is where the system should encode the conceptual minimum for what an order email must yield. If the golden allows obviously basic business fields to stay null, the run can look green while still missing the core order semantics we care about.

The architecture now enforces this directly. Generated shop-local tests and auto-golden generation both call the same visibility-based rule, which now also includes visible order_id. And for proof runs, the local profile now gets a dedicated proof_requirements.json plus a shared minimum-payload test, so the claimed email cannot go locally green while the fields the proof explicitly asks for are still null. Re-checking the Decathlon proof-local repo under this new gate now fails on both visible order_id and the proof-minimum payload.

Level 2 — Next Eligible Email By Predeclared Rule Built

The next strengthening step is to remove hand-picking. The selection rule must be declared before the run, and the next eligible email matching that rule becomes the proof case. That rule should be frozen into the proof artifacts themselves so the run can show what counted as eligible before the candidate was evaluated. Level 2 is now closed: the first eligible AliExpress failure under the frozen rule was selected automatically, same-run local Codex inference converged, the graph was redeployed cleanly, and the replayed AWS record returned a useful parsed payload.

Open Gaps

  • The runtime-gap assessment rule is now important as an operating guardrail, because higher proof levels increase the chance of ambiguous failures across merchants.
  • The next architectural complexity is within-shop variation, not shop isolation itself: multiple real email families for one merchant can still compete for the same extraction fields inside one shop graph.
  • The Decathlon run remains useful as a negative control: it showed the proof loop can fail honestly when the email body itself cannot satisfy the minimum useful-payload bar.
  • The current evidence is stronger because it now shows same-run convergence, the direct Codex transcript, and the replayed AWS payload without manual rescue or hidden state leakage.
  • The higher proof levels remain open because the current program still depends on one real-email example rather than a non-cherry-picked sequence or repeated run.
